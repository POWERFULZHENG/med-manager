# 接口契约文档

## 1. 文档目的

本文档用于统一"家庭常用药管理系统"的前后端接口约定，明确请求路径、请求方法、请求参数、响应格式、错误码、分页格式与字段命名规范，确保前后端开发、测试与联调过程保持一致。

## 1.1 技术栈

本系统采用以下技术栈：

- **前端**：Vue3 + Vite + TypeScript + Pinia + Vue Router + Element Plus + Axios
- **后端**：Spring Boot + Spring Security + JWT + MySQL
- **数据库**：MySQL
- **鉴权**：JWT (JSON Web Token)
- **接口风格**：RESTful API
- **数据格式**：JSON

---

## 2. 接口设计原则

1. 统一使用 REST 风格接口。
2. 统一使用 JSON 作为主要数据交互格式。
3. 所有接口必须有明确的请求方法与路径。
4. 所有响应必须使用统一返回体。
5. 所有列表接口必须支持分页与条件筛选。
6. 所有业务错误必须返回统一错误码。
7. 所有敏感接口必须进行鉴权。
8. 接口变更必须同步更新本文档。

---

## 3. 通用约定

### 3.1 基础路径

后端接口统一前缀：

```text
/api
```

示例：

- `/api/auth/login`
- `/api/medicines`
- `/api/stocks/in`

### 3.2 请求头

登录后请求必须携带以下请求头：

```http
Authorization: Bearer <token>
Content-Type: application/json
```

上传文件时：

```http
Content-Type: multipart/form-data
```

### 3.3 统一返回格式

所有接口统一返回如下结构：

```json
{
  "code": 0,
  "msg": "success",
  "data": {}
}
```

### 3.4 分页返回格式

所有分页列表统一返回如下结构：

```json
{
  "code": 0,
  "msg": "success",
  "data": {
    "pageNum": 1,
    "pageSize": 10,
    "total": 100,
    "records": []
  }
}
```

### 3.5 时间格式

统一使用 ISO 风格日期字符串：

- 日期：`YYYY-MM-DD`
- 日期时间：`YYYY-MM-DD HH:mm:ss`

示例：

```json
{
  "expireDate": "2026-12-31",
  "createdAt": "2026-04-22 10:30:00"
}
```

### 3.6 命名风格

- 请求字段使用 `camelCase`
- 响应字段使用 `camelCase`
- 数据库字段使用 `snake_case`
- 接口路径使用小写英文和短横线风格，当前项目优先保持简洁路径

## 4. 统一错误码

| code | 含义 |
|---|---|
| 0 | 成功 |
| 40001 | 参数错误 |
| 40002 | 登录失败 |
| 40003 | 未登录或 token 无效 |
| 40004 | 权限不足 |
| 40005 | 资源不存在 |
| 40006 | 数据重复 |
| 50000 | 系统错误 |
| 50001 | 业务失败 |
| 50002 | 数据校验失败 |

### 4.1 错误响应示例

```json
{
  "code": 40001,
  "msg": "参数错误",
  "data": null
}
```

## 5. 认证接口

### 5.1 用户登录

**POST** `/api/auth/login`

#### 请求体

```json
{
  "username": "admin",
  "password": "123456"
}
```

#### 响应示例

```json
{
  "code": 0,
  "msg": "success",
  "data": {
    "token": "jwt-token",
    "user": {
      "id": 1,
      "username": "admin",
      "nickname": "管理员",
      "roleCode": "ADMIN"
    }
  }
}
```

#### 说明
- 登录成功后返回 JWT token。
- 前端需保存 token，并在后续请求中统一携带。

## 6. 用户管理接口

### 6.1 用户列表

**GET** `/api/users`

#### 查询参数
- `pageNum`：页码，默认 1
- `pageSize`：每页数量，默认 10
- `keyword`：用户名或昵称模糊查询
- `status`：状态筛选
- `roleCode`：角色筛选

#### 响应示例

```json
{
  "code": 0,
  "msg": "success",
  "data": {
    "pageNum": 1,
    "pageSize": 10,
    "total": 2,
    "records": [
      {
        "id": 1,
        "username": "admin",
        "nickname": "管理员",
        "phone": "13800000000",
        "roleCode": "ADMIN",
        "status": 1,
        "createdAt": "2026-04-22 10:00:00"
      }
    ]
  }
}
```

---

### 6.2 新增用户

#### 请求体

```json
{
  "username": "user01",
  "password": "123456",
```

## 8. 库存管理接口

### 8.1 库存列表

**GET** `/api/stocks`

#### 查询参数
- `pageNum`
- `pageSize`
- `keyword`
- `expireStatus`
- `storageLocation`
- `medicineId`

#### expireStatus 约定
- `NORMAL`：正常
- `EXPIRING`：临期
- `EXPIRED`：过期

#### 响应字段建议
- `id`
- `medicineId`
- `medicineName`
- `batchNo`
- `quantity`
- `expireDate`
- `storageLocation`
- `warningDays`
- `status`
- `createdAt`
- `updatedAt`

---

### 8.2 入库

**POST** `/api/stocks/in`

#### 请求体

```json
{
  "medicineId": 1,
  "batchNo": "B20260401",
  "quantity": 10,
  "expireDate": "2026-12-31",
  "storageLocation": "药箱A",
  "warningDays": 30,
  "remark": "购药入库"
}
```

#### 业务约束
- `quantity` 必须大于 0。
- `expireDate` 不能为空。
- `batchNo` 建议唯一或在药品维度下唯一。

---

### 8.3 出库

**POST** `/api/stocks/out`

#### 请求体

```json
{
  "stockId": 1001,
  "quantity": 2,
  "remark": "服用两粒"
}
```

#### 业务约束
- 出库数量必须大于 0。
- 不能超过当前库存。
- 不允许扣减成负数。

---

### 8.4 库存调整

**PUT** `/api/stocks/{id}/adjust`

#### 请求体

```json
{
  "quantity": 5,
  "remark": "盘点调整"
}
```

#### 说明
- 调整后数量必须大于等于 0。
- 调整必须写入记录表。

---

### 8.5 临期药品列表

**GET** `/api/stocks/expiring`

#### 查询参数
- `pageNum`
- `pageSize`
- `days`：提前预警天数，默认按系统配置

---

### 8.6 过期药品列表

**GET** `/api/stocks/expired`

#### 查询参数
- `pageNum`
- `pageSize`
- `keyword`

## 10. OCR 识别接口

### 10.1 上传并识别

**POST** `/api/ocr/upload`

#### 请求类型
`multipart/form-data`

#### 请求字段
- `file`：图片文件

#### 响应示例

```json
{
  "code": 0,
  "msg": "success",
  "data": {
    "imageUrl": "/upload/xxx.jpg",
    "medicineName": "阿莫西林胶囊",
    "expireDate": "2026-12-31",
    "quantity": 1,
    "rawText": "..."
  }
}
```

#### 说明
- OCR 仅作为辅助识别。
- 识别结果不得直接写入库存。
- 前端必须展示识别结果供用户确认。

---

### 10.2 确认识别结果并入库

**POST** `/api/ocr/confirm`

#### 请求体

```json
{
  "imageUrl": "/upload/xxx.jpg",
  "medicineName": "阿莫西林胶囊",
  "specification": "0.5g*24粒",
  "quantity": 1,
  "expireDate": "2026-12-31",
  "storageLocation": "药箱A"
}
```

#### 说明
- 用户确认后才写入库存。
- 该接口应同步生成入库记录。

## 12. 文件上传约定

### 12.1 上传限制建议

- 图片格式：jpg、jpeg、png、webp
- 文件大小：建议不超过 10MB
- OCR 识别文件必须是清晰图片

### 12.2 上传返回

上传成功后建议返回：
- 文件访问地址
- 文件名
- 文件大小
- 上传时间

---

## 13. 字段校验规则

### 13.1 通用校验
- 必填字段不能为空。
- 数值字段必须符合业务范围。
- 日期字段必须合法。
- 字符串长度不得超过数据库限制。

### 13.2 业务校验
- 入库数量必须大于 0。
- 出库数量必须大于 0。
- 出库数量不得超过库存数量。
- 过期日期不得早于入库日期（如业务要求）。
- OCR 识别结果必须人工确认。

---

## 14. 接口兼容规则

1. 接口新增字段应保持向后兼容。
2. 不得随意删除已有字段。
3. 若必须修改字段含义，需同步更新版本与文档。
4. 接口废弃应保留过渡期。

---

2. 后端根据本文档生成 controller、dto、vo。
3. 任何字段名变更必须前后端同步确认。
4. 接口调通后再进行样式优化。
