# 目录结构文档

## 1. 文档目的

本文档用于统一“家庭常用药管理系统”的项目目录结构规范，确保前端、后端、文档、脚本、部署与自动化配置的组织方式清晰一致，便于 AI 智能体、开发人员与测试人员协同开发和维护。

---

## 2. 目录设计原则

1. **按职责分层**：不同职责的文件放在不同目录中。
2. **按业务聚合**：同一业务模块的页面、接口、状态、工具尽量集中管理。
3. **公共能力复用**：通用组件、工具、配置集中沉淀，避免重复实现。
4. **前后端分离**：前端、后端、文档、脚本、部署文件相互独立。
5. **便于扩展**：后续新增功能时，不破坏现有结构。
6. **便于自动化**：方便 CI/CD、测试、部署脚本自动识别与执行。

---

## 3. 仓库顶层目录结构

建议仓库根目录采用以下结构：

```text
med-manager/
├── frontend/                 # Vue3 前端项目
├── backend/                  # Spring Boot 后端项目
├── docs/                     # 设计文档、规范文档、接口文档
├── deploy/                   # 部署脚本、Nginx、Docker 配置
├── scripts/                  # 数据库脚本、初始化脚本、辅助脚本
├── .github/                  # GitHub Actions 配置
└── README.md                 # 项目总说明
```

### 3.1 顶层目录职责说明

- `frontend/`：前端源码与构建配置。
- `backend/`：后端源码与构建配置。
- `docs/`：架构文档、接口契约、数据库设计、规范说明。
- `deploy/`：部署环境相关配置，如 Nginx、Docker Compose、启动脚本。
- `scripts/`：数据库迁移脚本、初始化数据脚本、辅助运维脚本。
- `.github/`：CI/CD 工作流配置。
- `README.md`：项目简介、启动方式、部署说明、联系方式等。

## 4. 前端目录结构

### 4.1 前端标准结构

```text
frontend/
├── public/
├── src/
│   ├── assets/
│   ├── components/
│   ├── layouts/
│   ├── router/
│   │   ├── index.ts
│   │   ├── guards.ts
│   │   └── modules/
│   ├── store/
│   │   └── modules/
│   ├── api/
│   │   └── modules/
│   ├── views/
│   │   ├── auth/
│   │   ├── dashboard/
│   │   ├── medicine/
│   │   ├── stock/
│   │   ├── record/
│   │   ├── ocr/
│   │   └── user/
│   ├── hooks/
│   ├── utils/
│   ├── permissions/
│   ├── styles/
│   ├── types/
│   ├── App.vue
│   └── main.ts
├── .env.development
├── .env.production
├── index.html
├── package.json
└── vite.config.ts
```

### 4.2 前端目录职责说明

#### `public/`
用于存放无需打包处理的静态资源，如 favicon、固定图片等。

#### `src/assets/`
用于存放图片、图标、字体、背景图等静态资源。

#### `src/components/`
用于存放全局可复用组件，例如：
- 表格组件
- 表单组件
- 弹窗组件
- 文件上传组件
- 确认对话框组件

#### `src/layouts/`
用于存放系统整体布局组件，例如：
- 左侧菜单
- 顶部栏
- 主内容区域
- 后台默认布局

#### `src/router/`
用于存放路由配置与路由守卫：
- `index.ts`：路由入口
- `guards.ts`：登录鉴权、权限控制
- `modules/`：按业务拆分路由

#### `src/store/`
用于存放 Pinia 状态管理，主要保存：
- 用户信息
- token
- 菜单权限
- 全局 UI 状态

#### `src/api/`
用于存放接口请求封装，统一管理与后端交互。

#### `src/views/`
用于存放页面级组件，按业务模块组织。

#### `src/hooks/`
用于存放组合式函数，封装可复用逻辑。

#### `src/utils/`
用于存放工具函数，例如日期处理、下载文件、token 操作等。

#### `src/permissions/`
用于存放权限控制逻辑，如路由拦截、角色判断等。

#### `src/styles/`
用于存放全局样式和变量样式。

#### `src/types/`
用于存放 TypeScript 类型定义。

### 6.2 后端目录职责说明

#### `common/`
用于存放通用类，例如：
- 统一返回体
- 分页结果对象
- 自定义异常
- 全局异常处理
- 常量类
- 响应码枚举

#### `config/`
用于存放 Spring 配置类，例如：
- 跨域配置
- Web 配置
- MyBatis 配置
- Swagger / OpenAPI 配置

#### `security/`
用于存放登录鉴权与权限控制相关类，例如：
- JWT 工具类
- 认证过滤器
- 安全配置类
- 用户详情服务

#### `controller/`
用于存放接口入口层，只负责：
- 接收参数
- 参数校验
- 调用 service
- 返回结果

#### `service/`
用于存放业务接口定义。

#### `service/impl/`
用于存放业务实现类。

#### `mapper/`
用于存放数据库访问接口。

#### `entity/`
用于存放数据库实体类。

#### `dto/`
用于存放请求参数对象。

#### `vo/`
用于存放响应对象。

#### `enums/`
用于存放枚举类。

#### `utils/`
用于存放工具类。

#### `component/`
用于封装可复用的外部能力，如：
- 文件上传组件
- OCR 客户端组件

#### `resources/mapper/`
用于存放 MyBatis XML 文件（如采用 XML 映射方式）。

## 9. 环境划分策略

为确保系统的稳定性和安全性，建议至少划分以下三套环境：

### 9.1 开发环境（Development）
- **目的**：用于开发人员日常开发和调试
- **配置**：本地或开发服务器，数据库使用开发库
- **特点**：配置简单，便于调试，数据可以随时重置
- **访问地址**：http://localhost:3000（前端）/ http://localhost:8080（后端）

### 9.2 测试环境（Testing）
- **目的**：用于功能测试、回归测试和集成测试
- **配置**：独立的测试服务器，数据库使用测试库
- **特点**：配置接近生产环境，数据为测试数据，定期清理
- **访问地址**：http://test-med-manager.example.com

### 9.3 生产环境（Production）
- **目的**：正式上线运行的环境
- **配置**：高可用服务器，数据库使用生产库，定期备份
- **特点**：配置严格，安全措施完备，数据不可随意修改
- **访问地址**：http://med-manager.example.com

### 9.4 环境部署规范
- **代码管理**：使用 Git 分支管理，开发分支 -> 测试分支 -> 主分支
- **配置管理**：使用环境变量或配置文件区分不同环境的配置
- **部署流程**：开发完成 -> 提交代码 -> 自动化测试 -> 部署测试环境 -> 验收测试 -> 部署生产环境
- **监控与日志**：生产环境需配置监控和日志系统，及时发现和处理问题
- **环境隔离**：不同环境之间数据和配置严格隔离，确保测试不影响生产，生产数据安全

## 12. 命名规范

### 12.1 文件命名

- Vue 文件使用 PascalCase 或约定统一风格。
- TypeScript 文件使用小写英文或模块名命名。
- Java 类名使用 PascalCase。
- SQL 脚本建议使用版本号前缀。

### 12.2 目录命名

- 目录名统一使用小写英文。
- 不使用拼音。
- 不使用含义不明的缩写。

### 12.3 前后端命名规范

#### 前端命名规范
- **变量和函数**：使用 camelCase（驼峰命名法）
  - 示例：`userName`、`getMedicineList()`、`handleStockIn()`
- **组件名**：使用 PascalCase（帕斯卡命名法）
  - 示例：`MedicineList.vue`、`StockInForm.vue`、`OcrUpload.vue`
- **常量**：使用 UPPER_SNAKE_CASE（全大写下划线）
  - 示例：`API_BASE_URL`、`MAX_PAGE_SIZE`、`DEFAULT_PAGE_NUM`
- **类型和接口**：使用 PascalCase
  - 示例：`interface User`、`type MedicineStatus`、`interface ApiResponse`
- **文件名**：
  - Vue 组件：PascalCase（如 `MedicineList.vue`）
  - TypeScript 文件：camelCase 或 kebab-case（如 `medicine.ts`、`user-service.ts`）

#### 后端命名规范
- **类名**：使用 PascalCase
  - 示例：`MedicineController`、`UserService`、`StockEntity`
- **方法名**：使用 camelCase
  - 示例：`getMedicineList()`、`createStock()`、`updateUser()`
- **变量名**：使用 camelCase
  - 示例：`userName`、`medicineId`、`stockQuantity`
- **常量**：使用 UPPER_SNAKE_CASE
  - 示例：`MAX_PAGE_SIZE`、`DEFAULT_EXPIRE_DAYS`、`API_BASE_URL`
- **包名**：使用小写英文，点号分隔
  - 示例：`com.example.medmanager.controller`、`com.example.medmanager.service`
- **DTO/VO 类名**：使用 PascalCase，后缀标识类型
  - 示例：`MedicineCreateDTO`、`UserLoginVO`、`StockUpdateDTO`

#### 数据库命名规范
- **数据库名**：使用小写英文
  - 示例：`medmanager`
- **表名**：使用小写英文和下划线风格（snake_case）
  - 示例：`sys_user`、`sys_role`、`medicine_stock`、`inventory_record`
- **字段名**：使用小写英文和下划线风格（snake_case）
  - 示例：`user_name`、`medicine_id`、`expire_date`、`created_at`
- **主键**：统一使用 `id`
- **时间字段**：统一使用 `created_at`、`updated_at`
- **状态字段**：统一使用 `status`
- **布尔字段**：使用 `is_` 前缀
  - 示例：`is_active`、`is_deleted`

#### 接口命名规范
- **请求字段**：使用 camelCase
  - 示例：`{"userName": "admin", "medicineId": 1}`
- **响应字段**：使用 camelCase
  - 示例：`{"code": 0, "data": {"userName": "admin", "expireDate": "2026-12-31"}}`
- **接口路径**：使用小写英文和连字符（kebab-case）
  - 示例：`/api/medicines`、`/api/stocks/in`、`/api/users/{id}`

### 12.4 命名转换规则

- **前端 ↔ 后端**：字段名保持一致，都使用 camelCase
- **后端 ↔ 数据库**：需要转换
  - 后端 camelCase → 数据库 snake_case
  - 示例：`userName` → `user_name`，`expireDate` → `expire_date`
- **数据库 → 后端**：需要转换
  - 数据库 snake_case → 后端 camelCase
  - 示例：`user_name` → `userName`，`expire_date` → `expireDate`

## 13. 目录使用约束

1. `views/` 不放公共组件。
2. `components/` 不放业务页面。
3. `controller/` 不写复杂业务逻辑。
4. `service/` 不直接返回数据库实体给前端。
5. `docs/` 不放代码文件。
6. `scripts/` 不放业务源码。
7. `deploy/` 不放开发测试文件。

---

## 14. 推荐扩展原则

当项目新增功能时，应优先在原有模块中扩展，避免新建无必要的顶层目录。

例如：
- 新增“用药提醒”功能，应优先放入 `stock/` 或 `dashboard/` 相关模块。
- 新增“统计分析”功能，应优先放入 `dashboard/`。
- 新增“导入导出”功能，应放入对应业务模块中，而不是单独散落。
