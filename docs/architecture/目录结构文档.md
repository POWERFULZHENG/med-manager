# 目录结构文档

## 1. 文档目的

本文档用于统一"家庭常用药管理系统"的项目目录结构规范，确保前端、后端、文档、脚本、部署与自动化配置的组织方式清晰一致，便于 AI 智能体、开发人员与测试人员协同开发和维护。

---

## 2. 目录设计原则

1. **按职责分层**：不同职责的文件放在不同目录中。
2. **按业务聚合**：同一业务模块的页面、接口、状态、工具尽量集中管理。
3. **公共能力复用**：通用组件、工具、配置集中沉淀，避免重复实现。
4. **前后端分离**：前端、后端、文档、脚本、部署文件相互独立。
5. **便于扩展**：后续新增功能时，不破坏现有结构。
6. **便于自动化**：方便 CI/CD、测试、部署脚本自动识别与执行。

## 4. 部署目录结构（重要重构）

### 4.1 deploy/ 标准企业级结构

```text
deploy/
├── docker/
│   ├── backend.Dockerfile        # 后端镜像：JDK 17 Alpine
│   ├── frontend.Dockerfile       # 前端镜像：Node.js 24 + Nginx
│   └── docker-compose.yml        # 编排文件（deploy 目录使用）
├── nginx/
│   └── nginx.conf                # 反向代理 + 静态资源缓存配置
└── README.md                     # 详细部署操作手册
```

### 4.2 关键设计决策

| 决策 | 说明 |
|------|------|
| **Dockerfile 分离命名** | `backend.Dockerfile` / `frontend.Dockerfile`，避免同名文件混淆 |
| **双 docker-compose.yml** | 根目录 + `deploy/docker/` 都有，兼顾便利性和规范性 |
| **构建上下文统一** | 统一项目根目录，解决相对路径问题 |
| **源码零侵入** | 前端/后端 源码目录不包含任何部署文件 |

## 9. 新增文件/目录规范

| 文件类型 | 存放位置 |
|---------|---------|
| 新业务页面 | `src/views/{模块名}/` |
| 复用组件 | `src/components/` |
| 接口请求 | `src/api/{模块名}.ts` |
| 配置类 | `backend/.../config/` |
| 部署相关 | `deploy/{分类}/` |
| 数据库脚本 | `scripts/sql/V{版本号}_{描述}.sql` |
