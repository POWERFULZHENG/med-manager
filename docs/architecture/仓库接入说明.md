## 3. 仓库目录结构约定

仓库采用以下顶层结构：

```text
med-manager/
```

## 4. 仓库接入准备事项

在 AI 智能体或开发人员开始开发前，应先确认以下内容：

1. 仓库地址可访问。
2. 默认分支名称已明确。
3. 本地拉取权限已配置。
4. CI/CD 权限已配置。
5. 数据库连接信息已准备。
6. 外部 OCR 服务密钥已准备。
7. 前后端环境变量已拆分。
8. 目录结构与规范文档已同步到仓库。

---

## 5. 分支管理规范

### 5.1 分支职责

* `main`：生产稳定版本，仅允许发布合并。
* `develop`：日常集成分支。
* `feature/*`：功能开发分支。
* `hotfix/*`：线上紧急修复分支。
* `release/*`：发布候选分支（可选）。

### 5.2 分支规则

1. 所有功能必须先从 `develop` 拉出 `feature/*` 分支。
2. 所有开发完成后先合并回 `develop`。
3. 发布前从 `develop` 合并到 `main`。
4. 线上问题优先使用 `hotfix/*` 修复，再同步回 `develop`。

## 6. 权限与访问控制

### 6.1 仓库访问权限

建议按以下角色划分权限：

* **Owner / Maintainer**：可管理仓库、分支、权限、CI/CD。
* **Developer**：可推送开发分支、创建 PR、提交代码。
* **Read-only**：仅查看代码与文档。

### 6.2 AI 智能体权限

AI 智能体建议使用以下权限模式：

* 仅允许访问指定仓库。
* 不允许修改 `main` 分支。
* 必须通过 `feature/*` 分支提交。
* 不允许接触生产密钥与敏感配置。
* 仅允许读取必要的代码、文档与测试结果。

## 7. 环境变量与密钥管理

### 7.1 不允许写入仓库的内容

以下内容不得直接提交到仓库：

* 数据库真实密码
* JWT 密钥
* OCR 服务密钥
* 第三方平台 Token
* 生产环境 IP 白名单密钥
* 云存储 AccessKey / SecretKey

### 7.2 建议配置方式

使用以下方式进行管理：

* `.env.development`
* `.env.production`
* `.env.test`
* CI/CD 平台的 Secret 管理
* 服务器环境变量

### 7.3 示例

```env
VITE_API_BASE_URL=https://api-test.example.com
SPRING_PROFILES_ACTIVE=test
DB_USERNAME=medmanager
DB_PASSWORD=********
JWT_SECRET=********
OCR_APP_KEY=********
```

## 8. CI/CD 接入说明

### 8.1 GitHub Actions

建议在仓库中配置：

```text
.github/workflows/frontend-ci.yml
.github/workflows/backend-ci.yml
```

### 8.2 CI 触发规则

* `push` 到 `feature/*`：执行检查与测试。
* `push` 到 `develop`：执行完整构建并部署测试环境。
* `push` 到 `main`：执行生产发布流程。
* `pull_request`：执行代码审查和测试。

### 8.3 必须执行的检查

前端：

* ESLint
* TypeScript 检查
* 单元测试
* 构建

后端：

* 编译
* 单元测试
* 集成测试
* 打包

## 10. AI 智能体接入要求

如果由 AI 智能体参与开发，应遵守以下规则：

1. 先阅读仓库根目录下的规范文档。
2. 先确认目录结构，再开始写代码。
3. 每次改动只能聚焦一个功能模块。
4. 任何接口变更必须同步更新文档。
5. 任何数据库变更必须同步更新迁移脚本。
6. 任何发布前必须通过测试检查。
7. 不允许擅自新增未经批准的技术栈。

## 11. 提交规范

### 11.1 提交信息格式

建议采用如下格式：

```text
<type>: <summary>
```

示例：

* `feat: add medicine list page`
* `fix: correct stock deduction logic`
* `test: add stock service tests`
* `docs: update API contract`
* `refactor: extract common response class`

### 11.2 提交要求

* 每次提交只做一个主题。
* 提交内容必须可运行。
* 不要把格式化与业务变更混在一起。
* 关键修复必须附带测试说明。

---

## 12. 发布与回滚说明

### 12.1 发布流程

1. 代码合并到 `develop`。
2. 自动测试通过。
3. 生成构建产物。
4. 部署到测试环境。
5. 验证通过后合并到 `main`。
6. 部署到生产环境。

### 12.2 回滚要求

发布失败时必须能够：

* 回滚代码版本
* 回滚镜像版本
* 回滚配置版本
* 回滚数据库迁移（如支持）
