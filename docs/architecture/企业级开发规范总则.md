# 企业级开发规范总则

## 1. 文档目的

本规范用于指导项目按照企业级标准进行设计、开发、测试、部署与运维，确保系统具备以下能力：

- 可维护
- 可扩展
- 可测试
- 可审计
- 可回滚
- 可持续交付

适用于前后端分离项目、单体架构项目、模块化系统以及 AI 辅助开发场景。

---

## 2. 总体原则

所有开发活动必须遵循以下核心原则：

1. **规范优先**：先定义规则，再进行开发。
2. **稳定优先**：优先保证系统稳定性，而非短期速度。
3. **兼容优先**：接口与数据变更优先考虑向后兼容。
4. **最小变更**：每次只做必要改动，降低风险。
5. **自动验证**：能自动检查的内容，不依赖人工重复执行。
6. **全程留痕**：关键操作、关键变更必须可追踪。

---

## 3. 架构规范

### 3.1 架构治理

- 系统应采用清晰架构模式（单体、模块化单体、微服务需明确）。
- 模块边界必须清晰，禁止跨模块直接访问内部实现。
- 公共能力统一沉淀，避免重复开发。
- 新功能上线前需评估对现有架构的影响。

### 3.2 分层职责

推荐分层：

- Controller：接收请求、返回结果
- Service：业务逻辑处理
- Repository / Mapper：数据访问
- Domain / Entity：数据模型
- DTO / VO：输入输出对象

禁止：

- Controller 编写核心业务逻辑
- 页面直接访问数据库逻辑
- 多层重复实现同一逻辑

---

## 4. 代码规范

### 4.1 命名规范

- 类名使用 PascalCase
- 变量、函数使用 camelCase
- 常量使用 UPPER_SNAKE_CASE
- 包名、目录名使用小写英文
- 禁止拼音命名、无语义缩写

### 4.2 编码规范

- 单一职责原则：一个文件只做一类事情。
- 公共逻辑抽取复用。
- 避免硬编码。
- 异常处理统一规范。
- 所有新增代码需符合既有代码风格。

### 4.3 代码审查

每次提交前需检查：

- 是否符合架构规范
- 是否引入重复逻辑
- 是否破坏兼容性
- 是否有潜在安全风险
- 是否补充测试用例

---

## 5. 接口规范

### 5.1 接口设计

- 使用统一 API 风格（RESTful 推荐）。
- 请求路径命名统一。
- 接口版本可管理（如 `/api/v1`）。
- 请求参数、返回结构统一。

### 5.2 返回格式

统一结构示例：

```json
{
  "code": 0,
  "msg": "success",
  "data": {}
}
```

### 5.3 通用能力

所有列表接口建议支持：

- 分页
- 排序
- 条件筛选
- 模糊查询（如适用）

### 5.4 接口变更规范

- 接口变更必须同步更新文档。
- 尽量保持向后兼容。
- 废弃接口需给出迁移周期。

---

## 6. 数据规范

### 6.1 数据建模原则

- 主数据与业务数据分离。
- 日志数据与业务数据分离。
- 核心实体关系清晰。
- 数据结构预留扩展字段。

### 6.2 表设计要求

核心表建议包含：

- id
- created_at
n- updated_at
- created_by（可选）
- updated_by（可选）
- status（可选）

### 6.3 数据一致性

- 关键操作必须事务处理。
- 库存、金额、状态类数据必须防并发异常。
- 数据修改必须可追踪来源。

### 6.4 数据库变更

- 所有变更必须走迁移脚本。
- 禁止直接修改生产库结构。
- 发布前必须验证迁移脚本可执行。

---

## 7. 安全规范

### 7.1 认证授权

- 密码必须加密存储。
- 使用统一认证机制（JWT / Session / OAuth 等）。
- 权限控制遵循最小权限原则。

### 7.2 输入安全

- 所有输入做参数校验。
- 防 SQL 注入。
- 防 XSS。
- 防 CSRF（按架构场景）。

### 7.3 敏感信息管理

- 密钥、Token、密码不得写入代码仓库。
- 使用环境变量或密钥管理服务。
- 日志中禁止输出敏感信息。

---

## 8. 日志与审计规范

### 8.1 日志规范

日志分层：

- 应用日志
- 业务日志
- 错误日志
- 安全日志

日志建议包含：

- 时间
- 请求 ID
- 用户 ID（如适用）
- 操作类型
- 执行结果

### 8.2 审计要求

关键操作必须留痕：

- 登录 / 登出
- 数据新增 / 修改 / 删除
- 权限变更
- 导入导出
- 配置修改

---

## 9. 测试规范

### 9.1 测试层级

必须建立以下测试体系：

- 单元测试
- 集成测试
- 接口测试
- 冒烟测试
- 回归测试

### 9.2 测试要求

- 核心业务必须有自动化测试。
- Bug 修复应补充回归测试。
- 每次发布前执行关键路径验证。

---

## 10. CI/CD 规范

### 10.1 持续集成

代码提交后自动执行：

- 代码格式检查
- 静态扫描
- 单元测试
- 构建验证

### 10.2 持续部署

环境建议：

- 开发环境
- 测试环境
- 预发布环境
- 生产环境

部署要求：

- 发布物可追溯版本号。
- 支持快速回滚。
- 发布后自动健康检查。

---

## 11. 配置管理规范

- 配置按环境隔离。
- 不同环境使用独立配置文件或环境变量。
- 配置变更需记录版本。
- 敏感配置统一管理。

---

## 12. 性能规范

- 所有列表接口默认分页。
- 高频查询建立索引。
- 大文件上传设置大小限制。
- 外部接口调用设置超时与重试策略。
- 避免重复提交、重复消费。

---

## 13. 文档规范

项目必须维护以下文档：

- 架构设计文档
- 接口文档
- 数据库设计文档
- 部署文档
- 变更记录
- 故障处理手册

代码变更后，文档应同步更新。

---

## 14. AI 辅助开发规范

AI 参与开发时必须遵循：

1. 所有生成代码必须遵循项目既有结构。
2. 每次变更需说明影响范围。
3. 不允许擅自更换技术栈。
4. 不允许一次性大范围重构。
5. 生成代码后必须附测试建议。
6. 最终代码必须经过人工审核或自动门禁。

---

## 15. 发布与回滚规范

### 发布前检查

- 测试通过
- 文档更新
- 配置确认
- 数据库脚本确认
- 回滚方案确认

### 发布后检查

- 服务健康状态正常
- 核心接口正常
- 页面访问正常
- 错误日志无异常增长

### 回滚能力

必须支持：

- 代码版本回滚
- 镜像版本回滚
- 配置回滚
- 数据库回滚（如支持）

---

## 16. 企业级四项底线

任何项目变更不得破坏以下四项：

1. 可维护：结构清晰、职责明确。
2. 可测试：核心功能可自动验证。
3. 可追踪：问题能定位，操作有记录。
4. 可回滚：上线失败可快速恢复。

---

## 17. 总控执行口径（适用于 AI 智能体）

```text
所有开发活动必须满足企业级交付标准：
遵循架构规范、接口规范、数据规范、安全规范、测试规范与发布规范。
任何变更不得破坏系统稳定性、数据一致性、审计链路与回滚能力。
所有新增代码必须可维护、可测试、可追踪、可持续交付。
```
