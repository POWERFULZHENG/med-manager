# 家庭常用药管理系统

# 项目开发背景意义

考虑到平时每次看医生都开了许多药，家里堆满了很多种不同类型的药品，药箱子里的药凌乱，缺乏记录。每次去看病医生可能会开家里还有的药品，造成了资源。因此，为了能够能够解决这个问题，亟需一个家庭药物管理系统用于管理药物。

# 功能需求

## 登录功能

管理员和普通用户

## 药品入库和出库

手动输入或者拍照OCR识别；

## 库存管理

能够记录药品名称、数量和失效时间

# 系统架构

## 架构

前后端分离，前端使用Vue3，后端spring boot

## 安全风控

登录鉴权auth

## 最快实现方案

最佳最快速方案是：

1）先做最小可用版本（MVP）
只保留 5 个核心模块：
登录/权限、药品库存、入库出库记录、到期提醒、OCR识别。
先别做太多"高级功能"，例如复杂统计、智能推荐、批量导入、家庭成员协同这些都放到后面。这样最快能跑通完整业务闭环。你文件里的需求本身就已经足够支撑这个 MVP 范围。

2）后端直接用单体 Spring Boot
虽然你写的是前后端分离，但“最快实现”并不等于“分布式”。
建议后端先做成一个单体 REST API，结构清晰但不要拆微服务。目录按：
controller / service / repository / entity / dto / security / common
来分层即可。这样开发、调试、部署都最快。

3）前端用 Vue3 + Element Plus
原因很简单：
Element Plus 组件现成，表单、表格、弹窗、上传、日期选择器都能直接用。
这个系统本质上大量是“表格 + 表单 + 弹窗”，非常适合这种组合。
页面建议先做：
登录页、药品列表页、入库页、出库页、过期预警页、用户管理页。

4）OCR 不要自己做，先接接口
你需求里提到“手动输入或者拍照 OCR 识别”。
最快方案是：

前期只做“上传图片 → 调用第三方 OCR → 返回药品名称/有效期/数量候选项”

识别结果允许用户二次确认再入库
这样能极大降低开发量，也避免自己搞图像处理和模型训练。

5）权限设计尽量简单
按你文档里的“管理员和普通用户”即可。
建议最开始就定义两个角色：

管理员：管理药品、用户、库存、记录

普通用户：查看自己的家庭药品、进行基础出入库操作
这样权限控制非常直观。

6）数据库设计保持最少表
先建 5 张表就够了：

sys_user

sys_role

medicine

inventory_record

medicine_stock

其中最关键的是把药品基础信息和库存批次信息分开。因为同一种药可能有不同失效时间，不能只用一个"数量"字段硬扛。

7）如果你追求“最快上线”，推荐技术栈如下

前端：Vue3 + Vite + Element Plus + Axios

后端：Spring Boot + Spring Security

鉴权：JWT

数据库：MySQL

缓存：先不加 Redis，除非后面有验证码/会话/高频查询需求

文件存储：本地即可，后期再换 OSS

OCR：第三方 API

部署：Docker + Nginx

8）开发顺序
最省时间的顺序是：

登录与角色权限

药品列表与库存增删改查

入库/出库记录

到期时间提醒

OCR 上传识别

统计报表与优化

## 3）接口清单（最小可用版）

请求：

{

  "password": "123456"
}

### OCR识别

#### POST /api/ocr/upload

上传图片并识别。

返回：

{

### 最小实现的关键约定

为了后面不返工，建议一开始就定好这几条：

库存按批次管理，不要只用一个总数量字段

出库按批次扣减

所有库存变化都写记录表

临期规则统一：例如 expire_date - 当前日期 <= 30天

库存状态统一：NORMAL（正常）、EXPIRING（临期）、EXPIRED（过期）

OCR 只做辅助录入，不直接覆盖人工确认

权限只分管理员/普通用户，先别做复杂 RBAC

## 目录结构设计原则

1. 按“业务模块”组织，而不是按“技术堆砌”

不要把所有页面都放 pages/，所有接口都放 api/，所有组件都放 components/，最后越做越乱。
更好的方式是：围绕业务来分，比如 auth、medicine、stock、record、ocr、user。

这样做的好处是：

一个功能的页面、接口、状态、组件放在一起

后期新增“提醒”“统计”“家庭成员共享”时很容易扩展

代码阅读成本低

2. 控制“分层深度”，避免过度复杂

这个系统是中小型管理系统，不需要微服务式那种特别重的结构。
建议采用：

前端：views + components + api + store + utils

后端：controller + service + repository/mapper + entity + dto + security

即“清晰分层，但不过度分散”。

3. 公共能力统一沉淀

像下面这些内容，应该统一放公共目录：

登录鉴权

请求封装

响应格式

日期格式化

文件上传

常量枚举

异常处理

不要每个模块自己写一套。

4. 保持“页面文件薄、业务逻辑清”

页面只负责展示和交互，复杂逻辑放到：

前端：api / store / composables / utils

后端：service

这样后期维护最省事。

## 前端最佳目录结构

建议使用 Vue3 + Vite + Pinia + Vue Router + Element Plus，目录按业务模块拆分。

src/
├── assets/                 # 静态资源：图片、图标、样式资源
├── components/             # 全局通用组件
├── layouts/                # 布局组件：后台框架、侧边栏、头部
├── router/                 # 路由定义
├── store/                  # Pinia 全局状态
├── api/                    # 接口请求封装
├── utils/                  # 工具方法
├── hooks/                  # 组合式函数
├── views/                  # 页面级视图
│   ├── auth/               # 登录、注册
│   ├── dashboard/          # 首页仪表盘
│   ├── medicine/           # 药品管理
│   ├── stock/              # 库存管理
│   ├── record/             # 出入库记录
│   ├── ocr/                # OCR识别
│   └── user/               # 用户管理
├── styles/                 # 全局样式
├── permissions/            # 路由守卫、权限控制
└── main.ts

### 前端目录原则说明

1. views/ 放页面，不放零碎组件

例如：

views/medicine/MedicineList.vue

views/stock/StockIn.vue

views/ocr/OcrUpload.vue

页面就是页面，不要混杂太多可复用小组件。

2. components/ 放全局复用组件

例如：

BaseTable.vue

BaseForm.vue

ConfirmDialog.vue

UploadImage.vue

这些组件被多个模块复用，才放这里。

3. api/ 按模块拆文件

建议这样：

api/
├── auth.ts
├── medicine.ts
├── stock.ts
├── record.ts
├── ocr.ts
└── user.ts

每个文件只管自己模块的接口。

4. store/ 只存“跨页面共享状态”

比如：

用户登录信息

token

菜单权限

全局字典数据

不要把局部表单状态也塞进 store。

5. utils/ 放通用工具

比如：

日期格式化

下载文件

防抖节流

金额/数量格式化

临期判断辅助函数

### 前端推荐的模块级细化结构

views/

### 后端最佳目录结构

建议使用 Spring Boot + Spring Security + JWT + MyBatis-Plus / JPA。
目录建议按“分层 + 业务模块”结合来组织。

src/main/java/com/example/medmanager/
├── common/                 # 通用返回、异常、工具类、常量
├── config/                 # 配置类
├── security/               # 登录鉴权、JWT、权限控制
├── controller/             # 接口层
├── service/                # 业务接口
├── service/impl/           # 业务实现
├── mapper/                 # 数据访问层
├── entity/                 # 实体类
├── dto/                    # 请求/响应对象
├── vo/                     # 页面返回对象
├── enums/                  # 枚举
├── module/                 # 按业务聚合（可选）
│   ├── auth/
│   ├── medicine/
│   ├── stock/
│   ├── record/
│   ├── ocr/
│   └── user/
└── Application.java

### 后端目录原则说明

1. 控制层只做参数接收和结果返回

controller 不写复杂业务逻辑，只负责：

接收请求

参数校验

调用 service

返回统一结果

2. 业务逻辑统一放 service

比如：

入库时如何新增库存批次

出库时如何扣减库存

临期药品如何筛选

OCR 结果如何转换成药品数据

这些都必须放 service，别写到 controller 里。

3. entity 和 dto/vo 分开

不要直接把数据库实体原样返回前端。
建议：

entity：数据库映射

dto：前端请求入参

vo：后端返回给前端的数据

这样更安全，也更容易做字段裁剪。

4. common 统一放公共能力

建议包括：

Result / ApiResponse

BusinessException

GlobalExceptionHandler

PageResult

Constants

DateUtils

5. security 独立出来

登录鉴权是独立横切能力，建议单独放：

JWT 工具类

拦截器 / 过滤器

用户详情服务

权限注解

这样权限逻辑不会散落在各处。

### 后端推荐的业务模块化结构

如果你想让项目更整洁，建议按业务继续细分：

module/
├── auth/
│   ├── AuthController.java
│   ├── AuthService.java
│   ├── AuthServiceImpl.java
│   ├── LoginDTO.java
│   └── LoginVO.java
├── medicine/

├── record/
├── ocr/

这种方式适合项目后续迭代，尤其是你这个系统后面大概率还会加：

药品提醒

过期预警

家庭成员协同

用药记录

统计分析

模块化能减少返工。

### 这个项目最推荐的“落地原则”

结合你的需求，我建议你直接遵守这几条：

前端按业务模块拆目录

后端按 controller/service/mapper 分层

### 环境划分策略

为确保系统的稳定性和安全性，建议至少划分以下三套环境：

#### 1. 开发环境（Development）
- **目的**：用于开发人员日常开发和调试
- **配置**：本地或开发服务器，数据库使用开发库
- **特点**：配置简单，便于调试，数据可以随时重置
- **访问地址**：http://localhost:3000（前端）/ http://localhost:8080（后端）

#### 2. 测试环境（Testing）
- **目的**：用于功能测试、回归测试和集成测试
- **配置**：独立的测试服务器，数据库使用测试库
- **特点**：配置接近生产环境，数据为测试数据，定期清理
- **访问地址**：http://test-med-manager.example.com

#### 3. 生产环境（Production）
- **目的**：正式上线运行的环境
- **配置**：高可用服务器，数据库使用生产库，定期备份
- **特点**：配置严格，安全措施完备，数据不可随意修改
- **访问地址**：http://med-manager.example.com

#### 环境部署规范

#### 推荐路由设计原则

登录页独立，不要放进主布局里。

主系统统一一个布局 Layout，里面承载左侧菜单、顶部栏、内容区。

模块路由按业务拆分，比如 medicine、stock、record、ocr、user。

详情页和编辑页用参数路由，比如 /detail/:id。

临期、过期、入库、出库这些页面最好独立路由，方便权限控制和菜单展示。

各包职责

common：统一返回体、异常类、分页对象、通用常量

config：Spring 配置类、跨域配置、MyBatis 配置等

security：JWT、登录鉴权、权限拦截

controller：接口入口，只接收参数、调用 service、返回结果

service：业务接口

service.impl：业务实现

mapper：数据库访问层

entity：数据库实体类

dto：前端请求参数对象

vo：返回给前端的数据对象

enums：枚举类

utils：工具类

component：可复用的功能组件，比如 OCR 客户端、文件上传组件

#### 包名命名原则

包名用英文小写，不要中英文混杂。

不要频繁用缩写，比如 med 可以接受，但 mgt、sysx 这类尽量少用。

模块名和业务名一致，后期别人一看就知道用途。

service.impl 固定保留，这是最常见、最容易维护的写法。

# AI 的总控指令

你正在开发“家庭常用药管理系统”，必须严格遵守以下规则：

1. 只允许使用指定技术栈：Vue3 + Vite + TypeScript + Pinia + Vue Router + Element Plus；Spring Boot + Spring Security + JWT + MySQL。

2. 项目采用前后端分离、模块化单体架构，不允许擅自引入微服务、Redis、MQ、GraphQL 等未批准技术。

3. 前端必须按业务模块组织：auth、dashboard、medicine、stock、record、ocr、user。

4. 后端必须遵循 controller / service / mapper / entity / dto / vo / common / security 分层。

5. controller 只能接收参数和返回结果，不允许写业务逻辑。

6. 所有库存必须按批次管理，药品基础信息与库存批次必须分离。

7. OCR 只能做辅助录入，识别结果必须人工确认后才能写入库存。

8. 所有接口必须使用统一响应体 Result，统一错误处理，统一分页格式。

9. 每次改动只能聚焦一个功能模块，输出前必须列出修改文件、影响范围和自检结果。

10. 若发现已有架构或接口规范冲突，优先维护原规范，不得自行改动整体架构。

输出代码前，请先说明你将要修改的文件、接口和数据结构，再开始生成代码。
