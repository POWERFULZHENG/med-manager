# 数据库设计文档

## 1. 文档目的

本文档用于统一“家庭常用药管理系统”的数据库设计标准，明确核心表结构、字段含义、表间关系、索引策略、约束规则与建模原则，确保系统具备可维护、可扩展、可审计的数据基础。

---

## 2. 数据库设计原则

1. 药品基础信息与库存批次信息必须分离。
2. 库存必须按批次管理，不能只保存总数量。
3. 所有库存变化必须留痕，必须有出入库记录。
4. OCR 识别记录只用于审计，不直接作为库存主数据。
5. 用户、角色、药品、库存、记录之间的关系必须清晰。
6. 表结构应预留扩展空间，避免频繁重构。
7. 所有核心表应包含创建时间与更新时间。
8. 所有重要变更应通过迁移脚本管理。

---

## 3. 数据库命名规范

- 数据库名使用小写英文，建议采用：`medmanager`
- 表名使用小写英文和下划线风格
- 字段名统一使用小写英文和下划线风格
- 主键统一使用 `id`
- 时间字段统一使用 `created_at`、`updated_at`
- 逻辑状态字段统一使用 `status`

---

## 4. 核心数据表

系统核心表包括：

1. 角色表 `sys_role`
2. 用户表 `sys_user`
3. 药品基础表 `medicine`
4. 库存批次表 `medicine_stock`
5. 出入库记录表 `inventory_record`
6. OCR 识别记录表 `ocr_record`

---

## 5. 表结构设计

## 5.1 角色表 `sys_role`

用于定义系统角色，当前最小场景可支持管理员与普通用户。

| 字段 | 类型 | 约束 | 说明 |
|---|---|---|---|
| id | BIGINT | PK, AUTO_INCREMENT | 主键 |
| role_code | VARCHAR(20) | NOT NULL, UNIQUE | 角色编码，如 ADMIN、USER |
| role_name | VARCHAR(50) | NOT NULL | 角色名称 |
| remark | VARCHAR(200) | NULL | 备注 |
| created_at | DATETIME | NOT NULL | 创建时间 |
| updated_at | DATETIME | NOT NULL | 更新时间 |

### 建议索引
- `uk_sys_role_role_code`

### 数据示例
- `ADMIN`：管理员
- `USER`：普通用户

---

## 5.2 用户表 `sys_user`

用于保存系统登录用户信息与角色关联。

| 字段 | 类型 | 约束 | 说明 |
|---|---|---|---|
| id | BIGINT | PK, AUTO_INCREMENT | 主键 |
| username | VARCHAR(50) | NOT NULL, UNIQUE | 用户名 |
| password | VARCHAR(100) | NOT NULL | 密码哈希 |
| nickname | VARCHAR(50) | NOT NULL | 昵称 |
| phone | VARCHAR(20) | NULL | 手机号 |
| role_id | BIGINT | NOT NULL, FK | 角色ID |
| status | TINYINT | NOT NULL DEFAULT 1 | 用户状态：1启用，0禁用 |
| created_at | DATETIME | NOT NULL | 创建时间 |
| updated_at | DATETIME | NOT NULL | 更新时间 |

### 建议索引
- `uk_sys_user_username`
- `idx_sys_user_role_id`
- `idx_sys_user_status`

### 外键关系
- `role_id -> sys_role.id`

### 说明
- 密码必须加密存储，禁止明文保存。
- `status` 用于控制账号是否可登录。

---

## 5.3 药品基础表 `medicine`

用于保存药品的静态基础信息。

| 字段 | 类型 | 约束 | 说明 |
|---|---|---|---|
| id | BIGINT | PK, AUTO_INCREMENT | 主键 |
| medicine_name | VARCHAR(100) | NOT NULL | 药品名称 |
| specification | VARCHAR(100) | NULL | 规格 |
| manufacturer | VARCHAR(100) | NULL | 生产厂家 |
| unit | VARCHAR(20) | NOT NULL | 单位，如盒、瓶、片 |
| dosage_form | VARCHAR(50) | NULL | 剂型 |
| remark | VARCHAR(200) | NULL | 备注 |
| created_at | DATETIME | NOT NULL | 创建时间 |
| updated_at | DATETIME | NOT NULL | 更新时间 |

### 建议索引
- `idx_medicine_name`
- `idx_medicine_manufacturer`

### 说明
- 该表只保存药品基础信息，不保存具体库存数量。
- 同一种药品可对应多个库存批次。

---

## 5.4 库存批次表 `medicine_stock`

用于记录药品批次库存，是系统最核心的数据表之一。

| 字段 | 类型 | 约束 | 说明 |
|---|---|---|---|
| id | BIGINT | PK, AUTO_INCREMENT | 主键 |
| medicine_id | BIGINT | NOT NULL, FK | 药品ID |
| batch_no | VARCHAR(50) | NOT NULL | 批次号 |
| quantity | INT | NOT NULL DEFAULT 0 | 当前库存数量 |
| expire_date | DATE | NOT NULL | 失效日期 |
| storage_location | VARCHAR(100) | NULL | 存放位置 |
| warning_days | INT | NOT NULL DEFAULT 30 | 预警天数 |
| status | VARCHAR(20) | NOT NULL DEFAULT 'NORMAL' | 库存状态：NORMAL / EXPIRING / EXPIRED |
| created_at | DATETIME | NOT NULL | 创建时间 |
| updated_at | DATETIME | NOT NULL | 更新时间 |

### 建议索引
- `idx_stock_medicine_id`
- `idx_stock_batch_no`
- `idx_stock_expire_date`
- `idx_stock_status`
- `idx_stock_storage_location`

### 外键关系
- `medicine_id -> medicine.id`

### 业务约束
- 同一种药品可以有多个库存批次。
- 一个批次对应一个失效日期。
- 出库时应优先扣减临期或更早入库的批次（建议策略）。
- `quantity` 不得小于 0。

### 状态说明
- `NORMAL`：正常
- `EXPIRING`：临期
- `EXPIRED`：过期

### 临期判断建议
- 若 `expire_date - 当前日期 <= warning_days` 且未过期，则为临期。
- 若 `expire_date < 当前日期`，则为过期。

---

## 5.5 出入库记录表 `inventory_record`

用于记录所有库存变化，保证业务可追踪、可审计。

| 字段 | 类型 | 约束 | 说明 |
|---|---|---|---|
| id | BIGINT | PK, AUTO_INCREMENT | 主键 |
| stock_id | BIGINT | NOT NULL, FK | 库存批次ID |
| medicine_id | BIGINT | NOT NULL, FK | 药品ID |
| operator_id | BIGINT | NOT NULL, FK | 操作人ID |
| record_type | VARCHAR(20) | NOT NULL | 记录类型：IN / OUT / ADJUST |
| quantity_change | INT | NOT NULL | 变更数量 |
| before_quantity | INT | NOT NULL | 变更前数量 |
| after_quantity | INT | NOT NULL | 变更后数量 |
| remark | VARCHAR(200) | NULL | 备注 |
| created_at | DATETIME | NOT NULL | 操作时间 |

### 建议索引
- `idx_record_stock_id`
- `idx_record_medicine_id`
- `idx_record_operator_id`
- `idx_record_type`
- `idx_record_created_at`

### 外键关系
- `stock_id -> medicine_stock.id`
- `medicine_id -> medicine.id`
- `operator_id -> sys_user.id`

### 业务约束
- 每次入库、出库、调整都必须生成一条记录。
- `before_quantity` 与 `after_quantity` 必须真实反映库存变化。
- `quantity_change` 入库为正，出库通常为负，也可由业务统一定义后约束。

---

## 5.6 OCR 识别记录表 `ocr_record`

用于保存 OCR 识别过程，便于排查识别问题与审计。

| 字段 | 类型 | 约束 | 说明 |
|---|---|---|---|
| id | BIGINT | PK, AUTO_INCREMENT | 主键 |
| user_id | BIGINT | NOT NULL, FK | 操作人ID |
| image_url | VARCHAR(255) | NOT NULL | 图片地址 |
| ocr_result_json | LONGTEXT | NULL | OCR 原始识别结果 |
| status | TINYINT | NOT NULL DEFAULT 1 | 状态：1成功，0失败 |
| created_at | DATETIME | NOT NULL | 识别时间 |

### 建议索引
- `idx_ocr_user_id`
- `idx_ocr_created_at`

### 外键关系
- `user_id -> sys_user.id`

### 说明
- OCR 结果仅作辅助信息，不直接作为库存主数据。
- 最终入库必须经过人工确认。

---

## 6. 表之间的关系

### 6.1 关系说明

- `sys_role (1) -> (N) sys_user`
- `medicine (1) -> (N) medicine_stock`
- `medicine_stock (1) -> (N) inventory_record`
- `sys_user (1) -> (N) inventory_record`
- `sys_user (1) -> (N) ocr_record`

### 6.2 关系图文字说明

- 一个角色对应多个用户。
- 一个药品对应多个库存批次。
- 一个库存批次可产生多条出入库记录。
- 一个用户可产生多条库存操作记录和 OCR 识别记录。

---

## 7. 字段与约束说明

### 7.1 通用字段

所有核心业务表建议统一包含：
- `id`
- `created_at`
- `updated_at`

如需要审计，可扩展：
- `created_by`
- `updated_by`

### 7.2 状态字段

建议统一使用如下状态字段约定：

- 用户状态：`0` 禁用，`1` 启用
- 库存状态：`NORMAL`、`EXPIRING`、`EXPIRED`
- 记录类型：`IN`、`OUT`、`ADJUST`
- OCR 状态：`0` 失败，`1` 成功

### 7.3 唯一性建议

建议设置以下唯一约束：
- `sys_role.role_code`
- `sys_user.username`
- `medicine_stock.batch_no`（如业务允许，也可设为药品维度联合唯一）

---

## 8. 索引设计原则

1. 高频查询字段必须建索引。
2. 模糊查询字段可视数据量决定是否建立普通索引或全文索引。
3. 关联字段必须建索引。
4. 时间范围筛选字段建议建索引。
5. 不要对低选择性字段滥建索引。

### 8.1 推荐重点索引

- 用户名索引
- 药品名称索引
- 批次号索引
- 失效日期索引
- 记录时间索引
- 操作人索引

---

## 9. 数据初始化建议

### 9.1 基础角色数据

建议初始化以下角色：

- `ADMIN`：管理员
- `USER`：普通用户

### 9.2 管理员账号

建议初始化一个系统管理员账号，密码必须是加密后的哈希值。

### 9.3 初始化顺序

1. 创建数据库。
2. 创建角色表。
3. 插入角色数据。
4. 创建用户表并插入管理员账号。
5. 创建药品表。
6. 创建库存表。
7. 创建出入库记录表。
8. 创建 OCR 记录表。

---

## 10. 数据迁移规范

1. 所有数据库变更必须通过 SQL 迁移脚本管理。
2. 迁移脚本应按版本命名，例如：
   - `V001__init_schema.sql`
   - `V002__init_data.sql`
3. 数据库变更前必须先在测试环境验证。
4. 重要变更需准备回滚脚本。

---

## 11. 设计注意事项

### 11.1 不要把药品表与库存表合并
药品基础信息与库存批次信息职责不同，必须分离，否则后续会在批次管理、临期预警、出入库追踪上产生问题。

### 11.2 不要省略记录表
如果没有出入库记录表，系统只能看到当前库存，无法追踪谁在什么时候改了什么。

### 11.3 不要让 OCR 直接改库存
OCR 结果不稳定，必须先展示识别结果给用户确认，再写入库存。

### 11.4 不要忽略事务
库存变更与记录插入必须放在同一个事务中，避免数据不一致。
